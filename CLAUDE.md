# CLAUDE.md

Guidance for working in this repo. Read [SPEC.md](SPEC.md) for product intent and
[README.md](README.md) for build/run.

## Project

- iOS app «Прикорм» — a complementary-foods (solid foods) diary. SwiftUI + SwiftData,
  iOS 17+, bilingual RU/EN UI (Russian is the source language), no backend
  (food catalog is bundled JSON).
- Project is **generated by XcodeGen** from `project.yml`. Do not hand-edit
  `Prikorm.xcodeproj`; change `project.yml` and re-run `xcodegen generate`.
- Apple frameworks only — no third-party packages.

## Build / run convention

- The **user builds and runs the app himself**. Do NOT run `xcodegen`, `xcodebuild`,
  `simctl`, or open the simulator. Hand off build/run/test execution to the user.
- After adding or removing source files, remind the user to re-run `xcodegen generate`
  (the file list lives in `project.yml` via folder references).

## Architecture

- **Models** (`Sources/Models/`): SwiftData `@Model` types for user state — `Child`,
  `IntroductionStatus`, `FoodLog`. Plain value types — `Food`, `FeedingProfile`,
  enums in `Enums.swift`.
- **Services** (`Sources/Services/`): stateless business logic. `FoodCatalog`,
  `FeedingService`, `AllergenTracker`, `AllergenMaintenance`, `CalendarService`,
  `NotificationManager`.
- **Features** (`Sources/Features/`): SwiftUI views grouped by screen. Shared UI
  lives in `Features/Common/` — cartoon theme (`Theme`, `AppBackground`,
  `cartoonCard`), components (`BigButton`, `PillButton`, `ProgressRing`, `Chip`,
  `FoodIcon`), and the brand mascot **`Mascot`** (SwiftUI-drawn «Pudding»
  placeholder; `MascotMood` mood logic is unit-tested in `MascotTests`). Used only
  in emotional spots (empty states, onboarding, summary headers, success), never in
  working screens. App is locked to **light mode** (`RootView.preferredColorScheme(.light)`);
  there's no dark-theme palette. Display name is «Pudding» (`CFBundleDisplayName`);
  target/bundle id stay `Prikorm`/`com.prikorm`.
- **App-level constants** (`Sources/App/`): `AppLinks` (privacy/terms/support URLs —
  placeholders until the `site/` landing is deployed) and `Disclaimer` (single medical
  disclaimer text, reused in onboarding + profile for App Review 1.4.1). Methodology
  presets in `FeedingProfile` carry `source`/`sourceURL`/`caveat` (verified vs primary
  guidelines — see `docs/legal/methodology-sources.md`). Submission checklist:
  `docs/SUBMISSION.md`; test map: `docs/TEST-CASES.md`.
- **Keep business logic out of views.** Views call services; services are pure
  functions over `ModelContext` + catalog data. This is what makes the logic unit-testable
  without the UI.

## Localization (RU + EN)

- **String Catalog** `Resources/Localizable.xcstrings` — `sourceLanguage: ru`, English
  translations alongside. The catalog compiles automatically (Resources folder reference).
  `project.yml` sets `options.developmentLanguage: ru`. App follows the **device language**
  unless overridden in-app; «Союз педиатров РФ» preset is hidden outside `ru`
  (`FeedingProfile.visiblePresets`).
- **In-app language picker** (Profile → «Язык»): `AppLanguage` (system/ru/en) +
  `LanguageManager.apply` write the per-app `AppleLanguages` default; choice is stored in
  `@AppStorage("app.language")` and **applies on next launch** (iOS has no official
  runtime language-switch API, so the picker shows a "restart to apply" alert).
  `LanguageManager.apply(_:to:)` takes an injectable `UserDefaults` for testing.
- **Keys are the Russian strings** (so Xcode auto-extraction merges cleanly). Add new
  user-facing text → add the RU key + `en` value to the catalog.
- **How text gets localized (SwiftUI gotchas):**
  - `Text("литерал")`, `Button("…")`, `Label("…")`, `Section("…")`, `.navigationTitle("…")`
    auto-localize (literal → `LocalizedStringKey`). **No code change** beyond a catalog entry.
  - `Text(variable)` is **verbatim** — the variable must already be localized. Model strings
    (`enum.title`, `FeedingProfile.name/source/caveat`, `Food.localizedName`, `Disclaimer.medical`)
    return `String(localized:)`. Use `food.localizedName` in views, **never** raw `food.name`
    (which stays the canonical RU key for search/tests).
  - `Text(cond ? "A" : "B")` does **NOT** localize (ternary types as `String`) — wrap each
    branch in `String(localized:)`.
  - A `String` variable can't be passed to a `LocalizedStringKey` parameter. Components with
    **all-literal** callers take `LocalizedStringKey` (`BigButton`/`PillButton`/`GhostButton`,
    plus private helpers). Components with **mixed** callers (`Chip`, `StatusBadge`) keep
    `String`; wrap literal call-sites in `String(localized:)`.
  - **Collision:** «Поддержка» = Maintenance (log type) **and** Support (contact). Disambiguated
    by explicit keys `maintenance.type` / `support.link` (`String(localized:"…", defaultValue:"Поддержка")`).
  - **Plurals** live as `variations.plural` in the catalog (RU one/few/many, EN one/other) —
    e.g. `%lld продуктов введено`. No hand-rolled plural helper in code.
  - **Dates** follow the locale (no `.locale(.ru)` forcing). Dynamic keys (food names,
    weekday abbrevs) use `String(localized: String.LocalizationValue(key))`.
- **Tests are hosted** (`TEST_HOST` set), so `String(localized:)` resolves in the **simulator's
  language**. Keep equality assertions on **raw** values (`Food.name`, `DayEntry.foodName`,
  `Child.name`); for localized text assert non-empty or locale-robustly (see `DisclaimerTests`).
- **Test EN without changing the device:** in the scheme Test action set App Language = English,
  or launch with `-AppleLanguages "(en)" -AppleLocale en_US`. Screenshot both RU and EN.

## Testing conventions

- **Every task must include new/updated tests; all tests must pass before the next task.**
- Test target: `PrikormTests` (`Tests/`), wired in `project.yml`, uses
  `@testable import Prikorm` and runs via the `Prikorm` scheme's test action.
- One test file per service/model area: `ModelTests`, `FoodCatalogTests`,
  `FeedingServiceTests`, `AllergenTests`, `CalendarServiceTests`,
  `NotificationManagerTests`, `AppShellTests`.
- **SwiftData**: exercise `@Model` types and services through an **in-memory
  `ModelContainer`** (`ModelConfiguration(isStoredInMemoryOnly: true)`) — no disk, no app launch.
- **Determinism**: inject `Calendar` / `now:` into time-sensitive logic
  (e.g. `Child.ageInMonths(now:)`, `CalendarService.calendar`,
  `AllergenTracker.status(lastGiven:now:calendar:)` / `nextDue(lastGiven:calendar:)`,
  `AllergenMaintenance.now` / `.calendar`) so day-boundary / timezone / due-date math
  is reproducible. Don't read the system clock inside testable logic — services that
  aggregate over time (e.g. `AllergenMaintenance`) must thread an injected `now`/`calendar`
  into the tracker, never let it fall back to `Date()` / `Calendar.current`.
- **Notifications**: `NotificationManager` schedules through a `NotificationScheduling`
  protocol (a seam over `UNUserNotificationCenter`). Tests inject a mock center and
  assert on built `UNNotificationRequest`s — never touch the real notification center.
  The pure builder `requests(for:calendar:)` is tested directly. Note: `requestAuthorization()`
  intentionally goes straight to `UNUserNotificationCenter.current()` (the permission prompt
  is outside the seam); only the scheduling path (`apply` / `requests(for:)`) is mockable.
- Target ≥80% logic-layer coverage (services + models); UI is intentionally thin.

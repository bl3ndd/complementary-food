# Прикорм — ТЗ (черновик)

> Статус: планирование MVP. Документ живой, правим по ходу.
> Дата создания: 2026-06-20

## 1. Концепт

iOS-приложение — **дневник прикорма** для ребёнка.

Мы **трекер**, а не «что готовить». Никаких рецептов — только планирование введения,
отслеживание реакций и поддержка аллергенов. Это сознательно снижает медицинскую
ответственность на нас.

**Позиционирование (витрина vs движок):**
- **Витрина / ASO / название** = «дневник прикорма» — это понятный поисковый интент,
  по нему мамы реально ищут. Так нас находят.
- **Движок удержания под капотом** = трекер аллергенов — уникальная фича, которая
  держит юзера и после 12 мес (см. раздел «Продуктовая стратегия»).

**Главная фишка (то, чего нет у конкурентов):** после успешного введения аллерген
надо давать регулярно (раз в неделю+), иначе толерантность откатывается. Приложение
это отслеживает и напоминает.

## 2. Целевой пользователь

Родитель ребёнка ~4–12 месяцев, который начинает или ведёт прикорм. Хочет не
запутаться: что уже вводили, что в процессе, нет ли реакции, какие аллергены пора
«освежить».

## 3. Ключевые решения (зафиксировано)

- **Методика — на выбор пользователя** через «профиль прикорма» (пресеты + custom).
- **Один ребёнок** в MVP (но `Child` — отдельная сущность, чтобы v2 не требовал миграции).
- **iCloud-синхрон**: SwiftData + CloudKit.
- **Freemium / подписка** (StoreKit 2), гейтинг закладываем с первого дня.
- **Без рецептов** — совсем, даже в v2 (юридический риск).
- **Дисклеймер** «не медицинский совет, спроси педиатра» — на онбординге и в карточках аллергенов.

## 4. Core-логика

### 4.1 Профиль прикорма (FeedingProfile)

Методика = конфиг, а не хардкод. Весь core читает параметры отсюда.

Параметры:
- возраст старта (напр. 4 / 6 мес)
- длина окна наблюдения за новым продуктом (2–3 дня)
- частота поддержки аллергенов (1 или 2 раза/нед)
- список рекомендованных аллергенов

Пресеты: **WHO/ESPGHAN**, **US AAP/NIAID**, **Союз педиатров РФ**, **Custom** (custom — Pro).

### 4.2 Окно введения нового продукта

Новый продукт вводим по одному за раз.

Статусы продукта:
- `notIntroduced` — ещё не вводили
- `introducing` — в окне наблюдения (даём N дней, следим за реакцией)
- `introduced` — введён успешно
- `paused` — была реакция при первом вводе / отложили (можно повторить позже)
- `allergy` — подтверждённая аллергия (реакция случилась, когда продукт уже был введён).
  Напоминания выключаются, показываем предупреждение + совет обратиться к врачу.

### 4.3 Поддержка аллергенов (allergen maintenance) — главная механика

Топ-аллергены: яйцо, арахис, молочка, глютен/пшеница, рыба, морепродукты, соя,
орехи, кунжут.

После статуса `introduced` аллерген переходит в режим **поддержки**:
- трекаем `lastGivenDate` по каждому аллергену
- считаем статус по `allergenFrequencyPerWeek` из профиля:
  - 🟢 `ok` — давали в пределах окна
  - 🟡 `dueSoon` — скоро надо
  - 🔴 `overdue` — просрочено
- локальные пуш-напоминания, когда аллерген «протух»

**Поведение напоминаний (recurring weekly):** реализовано в `NotificationManager`
(`Sources/Services/NotificationManager.swift`).

- Разрешение на пуши просим **контекстно** — при вводе первого аллергена, не в
  онбординге (`requestAuthorization`).
- `refresh(context:profile:)` дёргается после каждого лога кормления/поддержки
  (из FoodDetailView и AllergensView). Он пересчитывает группы аллергенов через
  `AllergenMaintenance`, снимает устаревшие напоминания с префиксом `allergen-`
  и ставит новые.
- На каждую группу со статусом ≠ `ok` (т.е. `dueSoon`/`overdue`), которая введена и
  без флага аллергии, ставится **еженедельно повторяющееся** напоминание:
  `UNCalendarNotificationTrigger(dateMatching:, repeats: true)` с `weekday` от даты
  следующего приёма (`nextDue`) и временем 10:00. Так толерантность поддерживается
  без ручного перепланирования — триггер сам повторяется каждую неделю.
- Группы в статусе `ok` и группы с аллергией пропускаются (для последних напоминания
  гасятся — см. §4.4).
- Один запрос на группу; идентификатор `allergen-<rawValue группы>`, поэтому повторный
  `refresh` идемпотентно заменяет напоминание группы, а не плодит дубликаты.

### 4.4 Стейт-машина (черновик)

```
notIntroduced ──(начать ввод)──> introducing
introducing ──(прошло N дней, без реакции)──> introduced
introducing ──(реакция)──> paused
paused ──(повторить позже)──> introducing
introduced[аллерген] ──> режим поддержки (ok / dueSoon / overdue по lastGivenDate)
introduced (в поддержке) ──(реакция)──> allergy    // откат из поддержки + флаг «аллергия»
allergy ──(только по решению врача, ручной оверрайд)──> introducing
```

**Ключевая ветка:** реакция на УЖЕ введённый продукт = `allergy` (не `paused`).
Это другой случай, чем реакция при первом вводе: продукт раньше переносился нормально,
значит сигнал серьёзнее. Выходим из режима поддержки, гасим напоминания по этому
аллергену, показываем предупреждение. Вернуть в оборот можно только вручную (врач сказал).

## 5. Фичи

### MVP
- Профиль ребёнка (имя, дата рождения → авто-возраст + «можно начинать»)
- Онбординг с выбором профиля прикорма
- Каталог продуктов по категориям + флаг «аллерген» и группа
- Выбор продуктов для ввода → очередь/план
- Календарь введения + окно наблюдения
- Лог реакций (теги: сыпь/ЖКТ/дыхание/др. + заметка)
- Вкусовая оценка продукта: 😣 не понравилось / 😐 нейтрально / 😋 понравилось (отдельно от аллергии)
- Трекер поддержки аллергенов + напоминания
- iCloud-синхрон между своими устройствами
- Дисклеймеры

### v2+
- Несколько детей
- Экспорт дневника в PDF для педиатра
- Шеринг с партнёром
- Статистика/графики

## 7. Экраны

1. **Сегодня (Dashboard)** — что в окне наблюдения, какие аллергены пора дать, что нового
2. **Каталог продуктов** — фильтры по категории/аллергенам, поиск
3. **Карточка продукта** — статус, история, кнопка «дал сегодня»
4. **Календарь** — таймлайн введений по дням (см. §7.1)
5. **Аллергены** — статусы поддержки

7. **Онбординг** — профиль ребёнка + выбор методики + дисклеймер
8. **Paywall** — экран подписки

### 7.1 Календарь (ежедневный таймлайн введений)

Дневник по дням: что и когда давали, с типом (ввод/поддержка), реакцией и вкусовой
оценкой. Это «недостающая» четвёртая опора (food list / allergens / **calendar** /
напоминания).

- **CalendarView** — сетка/список дней; дни с активностью подсвечены. Источник
  подсветки — `CalendarService.activeDays()` (множество дат начала дня). Тап по дню → DayDetailView.
- **DayDetailView** — список продуктов за день: имя, тип (🆕 ввод / 🔁 поддержка),
  реакция и 😣/😐/😋 вкусовая оценка. Строится из `CalendarService.day(date)`.

Вся логика — в чистом сервисе `CalendarService` (`Sources/Services/CalendarService.swift`),
который группирует `FoodLog` по календарным дням (`days()` / `day(_:)` / `activeDays()`).
`Calendar` инъектируется, чтобы тесты проверяли границы дня и таймзоны детерминированно.
Внутри дня записи отсортированы по времени (ранние сверху), дни — новые сверху;
пустой день возвращает `DaySummary` без записей.

## 8. Данные (черновик моделей)

> CloudKit gotchas: все свойства опциональны/с дефолтом, нет `@Attribute(.unique)`,
> связи опциональны, дедуп в коде. Каталог продуктов из JSON НЕ синхроним (статика бандла),
> синхроним только пользовательские данные.

```
Child (id, name, birthDate)

FeedingProfile (id, name, startAgeMonths, observationDays,
                allergenFrequencyPerWeek, allergenList[], isPreset)

Food (id, name, category, isAllergen, allergenGroup, minAgeMonths)   // из JSON-бандла

IntroductionStatus (childId, foodId, status, introStartedAt, completedAt)

FoodLog (childId, foodId, date, type: intro|maintenance, reaction?, liking?, note?)
        // liking — вкусовая оценка: disliked|neutral|liked (😣/😐/😋), отдельно от аллергии
```

## 10. Стиль

Детский мультяшный: яркие цвета, скруглённые формы, дружелюбные иллюстрации
продуктов. Тёплый, не «медицинский». Светлая тема фиксирована (тёмная не
поддерживается — палитра захардкожена).

### Маскот «Pudding»

Бренд-маскот — мягкий пудинг-персонаж (`Sources/Features/Common/Mascot.swift`,
сейчас векторный SwiftUI-плейсхолдер до заказного арта). Меняет «настроение»
(`MascotMood`: neutral/happy/cheer/curious/sleepy/worried) под контекст.

Появляется **только в эмоциональных точках** — снять тревогу, дать бренду лицо:
пустые состояния, онбординг-гид, шапки-сводки (герой «Сегодня» реагирует на
прогресс, сводка аллергенов — на просрочку), момент успеха при вводе продукта.
В рабочих экранах (каталог, история, статусы) маскота НЕТ — там чистая взрослая
вёрстка. Маскот — акцент, а не вся UI.

«Pudding» — текущее рабочее имя приложения (`CFBundleDisplayName`); внутренний
таргет/bundle id пока `Prikorm`/`com.prikorm`.

## 11. Продуктовая стратегия

### JTBD (главный «найм»)
Родитель нанимает приложение не «вести список», а **«снять тревогу, что я делаю
прикорм неправильно и наврежу ребёнку»**. Продаём спокойствие, не таблицу.
Отсюда тёплый тон, дружелюбный дизайн, мягкий копирайтинг.

### Целевой сегмент
Ядро — **тревожные мамы первого ребёнка**: максимальная тревога → высокая готовность
платить, нет опыта → нужен поводырь, активны в мам-чатах → виральность.

### Риск №1: короткое окно жизни юзера
Прикорм — это ~4–12 мес, т.е. ~полгода жизни юзера, дальше встроенный churn и низкий LTV.
**Лечим трекером аллергенов:** поддержка аллергенов не заканчивается в 12 мес, это
единственная механика, продлевающая жизнь юзера за пределы окна прикорма.
→ ASO заходим как «дневник прикорма», удерживаем аллергенами.

### Метрики
- **North Star:** число «успешно поддержанных аллергенов в срок» в неделю.
- **Активация:** завёл ребёнка + ввёл 1-й продукт за первую сессию.
- **Aha-moment:** первое сработавшее напоминание про аллерген.
- **Retention-прокси:** % аллергенов в статусе 🟢 ok.
- **Деньги:** конверсия trial→paid на Custom-профиле.

### GTM / рост
- ASO на «прикорм», «дневник прикорма», «введение прикорма», «аллергены ребёнку».
- Педиатры как канал доверия (долго, но мощно).
- Мам-чаты / телеграм / инста-блогеры по ББ — там сидит ядро.
- Виральный крючок: экспорт дневника для педиатра (Pro) — юзер сам показывает апку.

### Что НЕ делаем в MVP
Статистика/графики, мультидети, соцфичи/сообщество. Святое, что остаётся даже при
жёсткой резке: ввод продукта + окно наблюдения + трекер аллергенов с напоминаниями.

## 12. Онбординг

**Флоу:**
0. **Welcome** — заставка, один CTA «Начать».
1. **Дисклеймер** ⚠️ — «не медицинский совет, согласуй с педиатром», чекбокс «Понятно». Обязателен.
2. **Профиль ребёнка** — имя (опц.) + дата рождения → авто-возраст. Обязателен.
3. **Выбор методики** — карточки пресетов (WHO/ESPGHAN, AAP/NIAID, Союз педиатров РФ) +
   🔒 Custom (Pro, ведёт на paywall). Можно сменить позже.
4. **Готово** — «Погнали!» → Dashboard.

**Решения:**
- Пуши **НЕ** просим в онбординге. Запрашиваем контекстно — при вводе первого аллергена
  (apple-style, выше согласий).
- Если возраст < возраста старта по методике → на Dashboard баннер «ещё рано, начни ~N мес».
- Custom-методика в онбординге = крючок на Pro (показываем, но залочено).

## 13. Пресеты методик ✅ VERIFIED

> ✅ Цифры сверены с первоисточниками (июнь 2026). Полная таблица «параметр →
> цитата → URL» — в [`docs/legal/methodology-sources.md`](docs/legal/methodology-sources.md).
> Где значение — клинический консенсус (LEAP/EAT), а не буква гайдлайна, это
> помечено в `FeedingProfile.caveat` и показывается в UI (App Review 1.4.1).

| Параметр | WHO / ESPGHAN | US AAP / NIAID | Союз педиатров РФ |
|---|---|---|---|
| Старт прикорма (мес) | 6 (ESPGHAN допускает с 4) | ~6 (4–6) | 4–6, оптимум 5–5.5 |
| Окно наблюдения (дни) | ~2–3 *(консенсус)* | ~2–3 *(консенсус)* | ~5–7, постепенный объём |
| Подход к аллергенам | ранний ввод | ранний, акцент арахис+яйцо | осторожный |
| Частота поддержки/нед | ~2 *(из EAT)* | ~2–3 (LEAP/EAT) | явно не указано |

**Честные оговорки:**
- Частота поддержки аллергенов — **клинический консенсус из LEAP/EAT, не официальная
  цифра гайдлайнов**. Надо оговаривать в самом приложении.
- РФ-методика: окно наблюдения длиннее, концепции «еженедельный аллерген» нет → наша
  киллер-фича слабее ложится на РФ-пресет.

**Итоги верификации (июнь 2026, сверено по первоисточникам):**
- **Старт прикорма** — прямые цифры источников: ВОЗ 6 мес, AAP ~6 мес, РФ 5 мес. ✅
- **Окно наблюдения** — у ВОЗ/ESPGHAN жёсткой цифры нет; «3 дня» — консенсус AAP/CDC
  (диапазон 3–5); РФ — «вводить за 5–7 дней». Защитимо как ориентир, помечено в `caveat`.
- **Частота поддержки аллергена** — клинический консенсус LEAP/EAT (EAT 2×/нед,
  LEAP 3×/нед, пинат-специфично); в программе РФ концепции нет вовсе. Помечено в `caveat`.
- **Набор аллергенов** — пресет США = FDA «Big 9»; ESPGHAN явно выделяет арахис+яйцо.

Детали и цитаты: [`docs/legal/methodology-sources.md`](docs/legal/methodology-sources.md).
